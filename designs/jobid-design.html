Job-Partitioned Async Design

// Apply → Store by Job ID → Trigger → SKILL.md Rules → Score → Shortlist
ASYNC · NOT REAL-TIME JOB_ID = PARTITION KEY SKILL.md = SCORING RULES
Architecture principle: When a candidate applies, we just store the CV under the Job ID — nothing else. Scoring is a separate async process triggered later (scheduled, manual, or threshold-based). This decouples ingestion from scoring, keeps ingestion cheap and fast, and lets you batch-score efficiently per job.

Storage Design — Job ID as Partition Key
Data Model
-- JOB table
jobs (
  job_id UUID PRIMARY KEY,
  title TEXT,
  jd_text TEXT, -- full JD
  skill_md_key TEXT, -- which SKILL.md
  status ENUM(OPEN,CLOSED),
  created_at TIMESTAMP
);

-- APPLICATION table (partitioned by job_id)
applications (
  app_id UUID PRIMARY KEY,
  job_id UUID REFERENCES jobs, ← partition key
  cv_blob_path TEXT, -- Azure Blob path
  cv_parsed JSONB, -- extracted fields
  score FLOAT, -- null until scored
  score_detail JSONB, -- per-dimension
  status ENUM(PENDING,SCORED,SHORTLISTED),
  applied_at TIMESTAMP
) PARTITION BY LIST (job_id);

-- Blob storage layout
blob://cvs/{job_id}/{app_id}.pdf
blob://skills/{category}/SKILL.md

SKILL.md — The Scoring Rulebook
---
name: resume-scorer-data-engineering
category: data-engineering
version: 1.2
scoring_model: claude-haiku-4
---

## Role
You are a senior technical recruiter scoring CVs for Data Engineering roles.
Score each CV against the provided JD on the dimensions below.
Return ONLY valid JSON. No prose.

## Scoring Dimensions
skills_match (30): Required skills from JD found in CV.
Partial credit for related tech (Spark vs Flink = 80%).

experience_relevance (25): Domain + seniority alignment.
Penalise if YoE < 70% of JD requirement.

impact_evidence (20): Quantified achievements, scale, outcomes.
"Led team" = 10pts. "Led team of 8, reduced latency 40%" = 20pts.

tech_stack_depth (15): Modern tooling, cloud, current.
Reward: Spark, dbt, Kafka, Airflow, vector DBs, LLM infra.

education_cert (10): Degree + relevant certifications.
AWS/GCP certs add 5pts each up to cap.

## Hard Disqualifiers
- FAIL if YoE < 50% of JD minimum
- FAIL if no relevant tech stack overlap
- FAIL if right-to-work flag = false

## Output Schema
{"score": 0-100, "pass": bool, "shortlist": bool,
 "dimensions": {...}, "flags": [...],
 "summary": "2 sentence max"}
How SKILL.md Gets Resolved
1. Job createdskill_md_key = "data-engineering"
2. Score triggeredload blob://skills/data-engineering/SKILL.md
3. Build promptSKILL.md body + JD text + CV text
4. Cache SKILL.mdRedis TTL 1hr — same prefix reused
5. Prompt cacheAnthropic caches static rubric prefix
💡 One SKILL.md per job category (not per job). "Data Engineering", "AI Engineer", "Product Manager" etc. All jobs of same category share the same rubric — HR can tune one file to affect all roles of that type.

Score Trigger Types
🕐
Scheduled Trigger
Cron job runs nightly. Picks all jobs with PENDING CVs. Scores in bulk overnight. Cheapest option.
nightly · low cost
📊
Threshold Trigger
Auto-fires when a job crosses N applicants (e.g. 100). Good for hot roles that fill fast.
event-driven
👆
Manual Trigger
Recruiter clicks "Score Now" in dashboard for a specific job. Useful for urgent or VIP roles.
on-demand
🔔
Deadline Trigger
Job closing date fires a final score of ALL pending CVs. Ensures nothing is missed before shortlist deadline.
date-based
Incremental Re-Scoring
Only PENDING CVs are scored each run. Already-scored CVs are skipped unless:

• SKILL.md version changed
• JD was updated by recruiter
• Recruiter explicitly requests full re-score

This keeps cost minimal on subsequent runs. A job open for 30 days only pays for each CV once.
